PAGE BREAKDOWN
Page Headers

HTML, CSS, JS

I created the page headers to be responsive to screen width, which promotes a consistent UX, where users can clearly tell which pages are on our website, and which are external. The header links to the main pages on the site (robot, members, sponsors, etc.) Using JavaScript, the header utilizes vertical space to allow for larger buttons on smaller screens, while keeping the client logo, navigation buttons, and slogan visible.

Landing Page

HTML, CSS, JS, PHP

The goal of the entire site, and by extension the landing page, was to closely resemble the website for team 2056. With that in mind, I made sure keep the design simple, using a basic CSS styling with flex grids for a mobile friendly experience. The part of this page I am most proud of myself for is the countdown timer. By using AJAX practices, I have the page send an asynchronous request to a PHP backend server, including the last known timings for events. The PHP server then pings 3 different API endpoints immediately before afterwards waiting for all 3 responses. The PHP backend can interpret the response from each endpoint, returning the data (if changed) to the front-end. If the data is unchanged, localStorage is used to get the timings.

Robots Page

HTML, CSS

The robots page is another simple page, with no fancy tricks going on. I made sure to use flex boxes again to allow for a mobile friendly experience, and filled the page with details of what the robot accomplished in 2025. Due to the clients wishes and the ongoing 2026 season, no details were added about the teams performance this year, to help them succeed at the provincial competition later this year.

Competition Scouting Page

HTML, CSS, JS, PHP, SQL

This is another page I am really proud of. I wanted to keep the page looking very simple and efficient, as it would be used almost exclusivey during chaotic competitions. The page only has a JS powered form on it, where users can enter details about robots they are watching at a competition. This form data is then sent to PHP, where the data is validated, cleaned, and moved to an SQL database. The database has a table for how each team performed in each match, and another table that simply keeps track of what match IDs correlate to each team, for quicker lookup and sorting by team captains when making decisions in the playoff stage of each competition. This was an extra feature not requested by the client, but as someone who was in charge of scouting for my own highschools team, I wish I had something like this to help us.

LANGUAGE BREAKDOWN
HTML

Header, Landing, Robot, Scouting

I implemented tags when possible, including divs, forms, buttons, and details tags. When creating the HTML for the header, I took special care to make sure that it wouldn't cause conflicts with other content below it in the page. I did my best to choose id and class names that are descriptive and easily identifiable when looking at either the HTML, CSS, or JavaScript code that uses any of the tags.

CSS

Header, Landing, Robot, Scouting

When creating the stylesheets for the website, I made sure to isolate as much of the styling as possible to not conflict with the stylingsheets being written by my team. I created a gloablStyle sheet, which ensures that y-overflow can be scrolled to, while eliminating the default margin around the body. I used a lot of flex boxes to allow for a mobile friendly experience, as well as many media tags to ensure that content could be resized or rearranged to be more readable on a mobile device.

JavaScript

Header, Landing, Scouting

Each component containing JavaScript has its own individual file, to promote seperation of concerns. The header JavaScript simply checks for the height of the button div, and then sets the image to the same height. Both of the scouting and timer JavaScript files utilize AJAX to send asynchronous commands to the PHP backend. The JS implemented for the timer utilizes localStorage to cache API results, which allows for faster requests, as if the response content hasn't changed, the API server can simply return a short HTML code, instead of sending all of the data for the request. The scouting page doesn't validate user input on the front-end, as many of the options are already locked down by HTML input types, however it does receive verification from the server that the contents are OK (not malicious), and displays a brief success or failure message to the user, before clearing the form for another submission.
